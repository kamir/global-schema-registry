## Design Principles

### 1. **Multi-Registry First**
- Registry selector visible on every page
- Clear indication of which registry is active
- Easy switching between registries

### 2. **Progressive Disclosure**
- Show high-level overview first
- Drill down for details on demand
- Avoid information overload

### 3. **Actionable Insights**
- Don't just show data; recommend actions
- Inline action buttons
- Guided workflows for complex tasks

### 4. **Safety & Confirmation**
- Confirm destructive actions
- Preview changes before applying
- Easy rollback options

### 5. **Performance**
- Lazy loading for large datasets
- Client-side filtering and sorting
- Efficient pagination

## User Workflows

### Workflow 1: Check Single Topic Compatibility

```
1. User navigates to "Compatibility â†’ Check Single Topic"
2. User selects registry from dropdown
3. User enters topic name or browses for it
4. User clicks "Check Compatibility"
5. System runs compatibility analysis

   - Per-version breakdown

   - User can click to immediately apply
8. If not compatible:
   - Show recommendations
   - User can export detailed report
```

### Workflow 2: Bulk Migration to FORWARD_FULL

```
1. User clicks "Migration Wizard" from dashboard
2. Wizard Step 1: Pre-Migration Analysis
   - User selects registries
   - User sets subject filter
   - System analyzes all matching subjects
3. Wizard Step 2: Review Results
   - System shows compatible/incompatible breakdown
   - User reviews migration plan
   - User decides which subjects to include
4. Wizard Step 3: Execute Migration
   - User confirms final migration
   - System creates backup
   - System migrates subjects one-by-one
   - Real-time progress shown
5. Wizard Step 4: Post-Migration Validation
   - System validates migration success
   - User reviews results
   - User can rollback if needed
6. User returns to dashboard
   - Dashboard reflects new compatibility modes
```

## API Integration

### API Calls from UI

```typescript
// Fetch compatibility overview
GET /api/v1/compatibility/overview?registry_id={id}

// Check single subject
POST /api/v1/compatibility/check
{
  "registry_id": "prod-confluent",
  "subject": "user-events-value",
  "check_type": "forward_full"
}

// Bulk compatibility check
POST /api/v1/bulk/check-compatibility
{
  "registry_ids": ["prod-confluent", "aws-glue"],
  "subject_filter": "user-*",
  "target_mode": "FORWARD_FULL"
}

// Set compatibility mode
PUT /api/v1/compatibility/{registry_id}/mode
{
  "mode": "FORWARD_FULL",
  "subject": "user-events-value"  // optional
}

// Migration wizard - analyze
POST /api/v1/migration/analyze
{
  "registry_ids": ["prod-confluent"],
  "subject_filter": "order-*",
  "target_mode": "FORWARD_FULL"
}

// Migration wizard - execute
POST /api/v1/migration/execute
{
  "analysis_id": "abc123",
  "subjects": ["order-events-value", "order-history-v1"],
  "create_backup": true
}
```
